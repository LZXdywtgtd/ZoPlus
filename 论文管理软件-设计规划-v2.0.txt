# 论文管理软件现代化项目设计规划
## Zotero Modernization with Tauri + AI（ZoPlus）
**版本号**：v2.0
**最后更新**：2026年6月12日

---

## 1. 项目概述
### 1.1 项目目标
打造一个媲美甚至超越 NoteExpress 的论文管理软件，具备现代化用户界面、卓越性能、极低内存占用，并深度集成人工智能智能功能。

### 1.2 核心策略
以 Zotero 为后端引擎，继承其强大的元数据识别能力、完整的 CSL 引用库和成熟的 Word 插件生态。采用 Tauri 技术栈完全重写前端界面，使用 Rust 语言处理所有性能敏感任务，并接入 MiniMax API 实现全方位 AI 增强。

### 1.3 最终成果
一个免费、开源、跨平台的桌面应用程序，满足学术研究人员的论文管理核心需求。应用内存占用控制在 50MB 以内，集成 AI 智能摘要、文献翻译、自动标签生成、毫秒级全文搜索、高级 PDF 标注等核心功能。

### 1.4 版本历史
| 版本 | 日期 | 主要变更 |
| ---- | ---- | -------- |
| v1.0 | 2026-06-09 | 初始设计规划 |
| v2.0 | 2026-06-12 | 新增"完全动态数据库自省系统"设计，重构数据库访问层 |

## 2. 前置环境要求
在开始项目开发前，开发环境必须满足以下条件：

### 2.1 基础开发环境
- **Node.js**：版本 ≥ 18.0.0（LTS 版本推荐）
- **Rust 工具链**：最新稳定版（通过 rustup 安装）
- **Tauri CLI**：最新稳定版
- **Git**：版本控制工具
- **代码编辑器**：推荐 VS Code 并安装 Tauri、Rust、TypeScript 相关插件

### 2.2 依赖软件
- **Zotero**：版本 ≥ 6.0（必须已安装并初始化数据库）
- **浏览器**：现代浏览器（用于前端开发调试）
- **操作系统**：Windows 10/11、macOS 12+ 或主流 Linux 发行版

### 2.3 第三方服务
- **MiniMax 开发者账号**：获取 API Key 用于 AI 功能集成
- **阿里云服务器**：2 核 2G 内存 3M 带宽 40GB 云盘（用于自定义云同步服务）

## 3. NoteExpress 功能对照与实现方案
| NoteExpress 功能 | 实现程度 | 实现方案 |
|----------------|---------|----------|
| **文献导入**（PDF、RIS、BibTeX） | ✅ 完整支持 | 继承 Zotero 原生支持并进行增强，通过 Zotero API 或直接读写 SQLite 数据库实现 |
| **文件夹/标签分类** | ✅ 完整支持 | 直接使用 Zotero 的 collections 和 tags 表结构 |
| **全文搜索**（标题、作者、摘要、PDF 全文） | ✅✅ 大幅超越 | 使用 Rust 生态的 Tantivy 搜索引擎替代 Zotero 的 SQLite FTS，实现毫秒级响应和精准中文分词 |
| **PDF 预览** | ✅ 完整支持 | 使用 pdf.js 在系统 WebView 中渲染 PDF 文档 |
| **PDF 标注**（高亮、笔记） | ✅✅ 超越 NoteExpress | 基于 pdf.js 实现矩形、椭圆、箭头、自由绘制、文本高亮等多种标注方式，标注数据以 JSON 格式存储 |
| **引用格式导出**（APA、GB/T 7714 等） | ✅ 完整支持 | 继承 Zotero 的 citeproc-js 引擎，支持 10,000+ 种引用格式 |
| **Word 插件** | ✅ 完整支持 | 直接使用 Zotero 原生 Word 插件，应用与 Zotero 共享数据库，插件功能不受影响 |
| **云同步** | ✅ 增强实现 | 基于阿里云服务器搭建自定义同步服务，同步数据库和附件文件 |
| **移动端支持** | ⚠️ 暂不实现 | 未来可基于 Tauri 移动版或独立 Flutter 应用开发 |
| **团队协作** | ⚠️ 暂不实现 | 未来可基于 Zotero Groups 功能进行增强 |
| **AI 智能摘要** | ✅✅ 独家亮点 | 调用 MiniMax API 生成文献摘要、翻译标题、自动生成分类标签 |
| **文献问答** | ✅ 独家亮点 | 提取 PDF 全文文本，使用 MiniMax 长文本 API 实现针对单篇文献的智能问答 |
| **自动元数据补全** | ✅ 增强实现 | 使用 MiniMax 大模型从 PDF 文本中高准确率提取标题、作者、DOI 等元数据 |

## 4. Zotero 缺点及修补方案
| Zotero 现有缺点 | 修补方案 | 技术实现思路 |
|-------------|----------|----------|
| **UI 界面陈旧（基于 XUL 技术）** | 完全重写现代化前端界面 | 使用 React + TypeScript 构建，搭配 Ant Design 或 Mantine 组件库，支持亮色/暗色模式 |
| **内存占用过高（约 200MB）** | 彻底替换运行时环境 | 使用 Tauri 替代 Electron，利用系统原生 WebView，将应用内存控制在 10~50MB 范围内 |
| **PDF 标注功能薄弱** | 实现专业级 PDF 标注系统 | 在 pdf.js 渲染层之上叠加 Canvas 绘制层，支持多种标注类型，标注数据独立存储 |
| **全文搜索速度缓慢** | 替换搜索引擎核心 | 集成 Rust 原生 Tantivy 搜索引擎，构建独立倒排索引，支持中文全文检索 |
| **缺乏原生 AI 能力** | 深度集成人工智能功能 | 在 Rust 后端统一调用 MiniMax API，所有 AI 功能内置，无需额外插件 |
| **PDF 元数据识别准确率低** | 大模型增强元数据提取 | 先提取 PDF 文本内容，再通过精心设计的提示词让 MiniMax 输出结构化元数据 |
| **官方云同步容量受限** | 提供自定义云同步选项 | 基于阿里云服务器搭建同步服务，支持数据库和附件的增量同步 |
| **插件系统复杂且不稳定** | 核心功能内置化 | 将常用功能直接写入应用核心，不依赖第三方插件系统 |
| **数据库结构硬编码** | 完全动态数据库自省 | 运行时扫描数据库表结构，动态构建 SQL，不依赖硬编码表名/字段名 |

## 6. 完全动态数据库自省系统
### 6.1 系统概述
ZoPlus v2.0 实现了**完全动态数据库自省系统**，彻底摒弃硬编码表名和字段名的做法。系统运行时自动扫描 Zotero 数据库的实际表结构，动态构建 SQL 语句，确保对不同版本 Zotero 数据库的兼容性。

### 6.2 核心组件

#### 6.2.1 DatabaseMetadata（数据库元数据中心）
```rust
pub struct DatabaseMetadata {
    tables: HashMap<String, TableMetadata>,  // 表名(小写) -> 表元数据
}

impl DatabaseMetadata {
    /// 扫描数据库所有表和字段
    pub fn scan_database(conn: &Connection) -> SqliteResult<Self>

    /// 检查表是否存在（大小写不敏感）
    pub fn table_exists(&self, name: &str) -> bool

    /// 获取表元数据（大小写不敏感）
    pub fn get_table(&self, name: &str) -> Option<&TableMetadata>

    /// 检查字段是否存在（大小写不敏感）
    pub fn column_exists(&self, table_name: &str, column_name: &str) -> bool

    /// 智能查找表（支持多候选表名）
    pub fn find_table<'a>(&self, candidates: &[&'a str]) -> Option<&'a str>
}
```

**核心特性**：
- 运行时扫描数据库所有表和字段元数据
- 大小写不敏感的表名/字段名查询
- 多候选表名支持（处理 Zotero 版本差异）
- 全局缓存机制，避免重复扫描

#### 6.2.2 TableMetadata（表元数据）
```rust
pub struct TableMetadata {
    pub name: String,                    // 表名
    pub columns: Vec<ColumnMetadata>,    // 字段列表
}
```

#### 6.2.3 ColumnMetadata（字段元数据）
```rust
pub struct ColumnMetadata {
    pub name: String,        // 字段名
    pub data_type: String,   // 数据类型
    pub is_primary_key: bool,// 是否主键
    pub is_nullable: bool,   // 是否可为空
}
```

#### 6.2.4 DynamicSqlBuilder（动态 SQL 构建器）
```rust
pub struct DynamicSqlBuilder<'a> {
    metadata: &'a DatabaseMetadata,
}

impl DynamicSqlBuilder {
    /// 构建 INSERT 语句（自动过滤不存在的字段）
    pub fn insert(&self, table: &str, data: &HashMap<&str, &str>) -> Option<(String, Vec<&str>)>

    /// 构建 SELECT 语句
    pub fn select(&self, table: &str, columns: &[&str], where_clause: &str) -> Option<String>

    /// 构建 UPDATE 语句
    pub fn update(&self, table: &str, data: &HashMap<&str, &str>, where_clause: &str) -> Option<(String, Vec<&str>)>

    /// 构建 DELETE 语句
    pub fn delete(&self, table: &str, where_clause: &str) -> Option<String>
}
```

**核心特性**：
- 自动检测表和字段是否存在
- 智能构建 INSERT/SELECT/UPDATE/DELETE 语句
- 只插入实际存在的列（避免 SQL 错误）

#### 6.2.5 ZoteroTableCandidates（Zotero 表名候选列表）
```rust
pub struct ZoteroTableCandidates;

impl ZoteroTableCandidates {
    /// 作者关联表（兼容不同 Zotero 版本）
    pub const CREATORS: &'static [&'static str] = &["itemCreators", "itemAuthors", "itemCreator"];

    /// 文献数据值表
    pub const ITEM_DATA_VALUES: &'static [&'static str] = &["itemDataValues"];

    /// 文献表
    pub const ITEMS: &'static [&'static str] = &["items"];

    /// 文献标签表
    pub const ITEM_TAGS: &'static [&'static str] = &["itemTags", "tags"];

    // ... 其他表候选
}
```

### 6.3 动态列名检测
Zotero 不同版本可能使用不同的列名，系统实现了动态检测功能：

#### 6.3.1 find_items_key_column（查找 items 表的 key 列）
```rust
/// 动态检测 items 表中的 key 列名
///
/// Zotero 可能使用 key、itemKey、keyString、itemKeyString 等列名
/// 此函数扫描 items 表的列结构，查找匹配的列
pub fn find_items_key_column(metadata: &DatabaseMetadata) -> Option<&str>
```

### 6.4 全局缓存机制
```rust
/// 获取缓存的元数据（如果已缓存则返回，否则扫描并缓存）
pub fn get_cached_metadata(conn: &Connection) -> SqliteResult<DatabaseMetadata>

/// 清除元数据缓存（数据库变更后调用）
pub fn invalidate_metadata_cache()
```

**缓存策略**：
- 首次调用时扫描数据库并缓存
- 后续调用直接返回缓存（验证表数量一致）
- 数据库变更后手动失效缓存

### 6.5 使用示例
```rust
use crate::db::{DatabaseMetadata, DynamicSqlBuilder, ZoteroTableCandidates, get_cached_metadata};

// 获取缓存的元数据
let metadata = get_cached_metadata(&conn)?;

// 查找表（支持多候选表名）
let creators_table = metadata.find_table(ZoteroTableCandidates::CREATORS);

// 构建动态查询
let sql = DynamicSqlBuilder::new(&metadata)
    .select("items", &["itemID", "key", "title"], "itemID = ?");

// 动态插入数据
let data = HashMap::from([
    ("title", "论文标题"),
    ("year", "2024"),
]);
if let Some((sql, params)) = DynamicSqlBuilder::new(&metadata).insert("items", &data) {
    // 执行 SQL...
}
```

## 9. 注意事项与约束
### 9.1 数据安全
- 初期严格采用只读方式访问 Zotero 数据库，避免数据损坏风险
- 所有写操作必须经过严格测试，优先写入自定义字段
- 定期备份 Zotero 数据库和应用数据
- 云同步服务必须实现数据加密传输和存储

### 9.2 性能要求
- 应用启动时间控制在 2 秒以内
- 文献列表滚动流畅，无明显卡顿
- 全文搜索响应时间小于 100 毫秒
- 内存占用始终控制在 50MB 以内
- PDF 文档加载和渲染流畅

### 9.3 兼容性要求
- 保持与 Zotero 数据库结构的完全兼容
- **通过动态自省系统支持不同版本的 Zotero 数据库**
- 确保 Zotero Word 插件能够正常工作
- 支持 Windows、macOS 和 Linux 三大平台
- 兼容主流的 PDF 文档格式

### 9.4 第三方服务使用
- MiniMax API 调用必须添加合理的错误处理和重试机制
- 为 AI 功能添加用量控制，避免超出 API 限额
- 阿里云服务器资源使用需进行合理规划和监控
- 所有第三方服务依赖必须可配置，方便用户替换
