# 论文管理软件现代化项目设计规划
## Zotero Modernization with Tauri + AI（ZoPlus）
**版本号**：v1.0  
**最后更新**：2026年6月9日

---

## 1. 项目概述
### 1.1 项目目标
打造一个媲美甚至超越 NoteExpress 的论文管理软件，具备现代化用户界面、卓越性能、极低内存占用，并深度集成人工智能智能功能。

### 1.2 核心策略
以 Zotero 为后端引擎，继承其强大的元数据识别能力、完整的 CSL 引用库和成熟的 Word 插件生态。采用 Tauri 技术栈完全重写前端界面，使用 Rust 语言处理所有性能敏感任务，并接入 MiniMax API 实现全方位 AI 增强。

### 1.3 最终成果
一个免费、开源、跨平台的桌面应用程序，满足学术研究人员的论文管理核心需求。应用内存占用控制在 30MB 以内，集成 AI 智能摘要、文献翻译、自动标签生成、毫秒级全文搜索、高级 PDF 标注等核心功能。

## 2. 前置环境要求
在开始项目开发前，开发环境必须满足以下条件：

### 2.1 基础开发环境
- **Node.js**：版本 ≥ 18.0.0（LTS 版本推荐）
- **Rust 工具链**：最新稳定版（通过 rustup 安装）
- **Tauri CLI**：最新稳定版
- **Git**：版本控制工具
- **代码编辑器**：推荐 VS Code 并安装 Tauri、Rust、TypeScript 相关插件

### 2.2 依赖软件
- **Zotero**：版本 ≥ 6.0（必须已安装并初始化数据库）
- **浏览器**：现代浏览器（用于前端开发调试）
- **操作系统**：Windows 10/11、macOS 12+ 或主流 Linux 发行版

### 2.3 第三方服务
- **MiniMax 开发者账号**：获取 API Key 用于 AI 功能集成
- **阿里云服务器**：2 核 2G 内存 3M 带宽 40GB 云盘（用于自定义云同步服务）

## 3. NoteExpress 功能对照与实现方案
| NoteExpress 功能 | 实现程度 | 实现方案 |
|----------------|---------|----------|
| **文献导入**（PDF、RIS、BibTeX） | ✅ 完整支持 | 继承 Zotero 原生支持并进行增强，通过 Zotero API 或直接读写 SQLite 数据库实现 |
| **文件夹/标签分类** | ✅ 完整支持 | 直接使用 Zotero 的 collections 和 tags 表结构 |
| **全文搜索**（标题、作者、摘要、PDF 全文） | ✅✅ 大幅超越 | 使用 Rust 生态的 Tantivy 搜索引擎替代 Zotero 的 SQLite FTS，实现毫秒级响应和精准中文分词 |
| **PDF 预览** | ✅ 完整支持 | 使用 pdf.js 在系统 WebView 中渲染 PDF 文档 |
| **PDF 标注**（高亮、笔记） | ✅✅ 超越 NoteExpress | 基于 pdf.js 实现矩形、椭圆、箭头、自由绘制、文本高亮等多种标注方式，标注数据以 JSON 格式存储 |
| **引用格式导出**（APA、GB/T 7714 等） | ✅ 完整支持 | 继承 Zotero 的 citeproc-js 引擎，支持 10,000+ 种引用格式 |
| **Word 插件** | ✅ 完整支持 | 直接使用 Zotero 原生 Word 插件，应用与 Zotero 共享数据库，插件功能不受影响 |
| **云同步** | ✅ 增强实现 | 基于阿里云服务器搭建自定义同步服务，同步数据库和附件文件 |
| **移动端支持** | ⚠️ 暂不实现 | 未来可基于 Tauri 移动版或独立 Flutter 应用开发 |
| **团队协作** | ⚠️ 暂不实现 | 未来可基于 Zotero Groups 功能进行增强 |
| **AI 智能摘要** | ✅✅ 独家亮点 | 调用 MiniMax API 生成文献摘要、翻译标题、自动生成分类标签 |
| **文献问答** | ✅ 独家亮点 | 提取 PDF 全文文本，使用 MiniMax 长文本 API 实现针对单篇文献的智能问答 |
| **自动元数据补全** | ✅ 增强实现 | 使用 MiniMax 大模型从 PDF 文本中高准确率提取标题、作者、DOI 等元数据 |

## 4. Zotero 缺点及修补方案
| Zotero 现有缺点 | 修补方案 | 技术实现思路 |
|-------------|----------|----------|
| **UI 界面陈旧（基于 XUL 技术）** | 完全重写现代化前端界面 | 使用 React + TypeScript 构建，搭配 Ant Design 或 Mantine 组件库，支持亮色/暗色模式 |
| **内存占用过高（约 200MB）** | 彻底替换运行时环境 | 使用 Tauri 替代 Electron，利用系统原生 WebView，将应用内存控制在 10~30MB 范围内 |
| **PDF 标注功能薄弱** | 实现专业级 PDF 标注系统 | 在 pdf.js 渲染层之上叠加 Canvas 绘制层，支持多种标注类型，标注数据独立存储 |
| **全文搜索速度缓慢** | 替换搜索引擎核心 | 集成 Rust 原生 Tantivy 搜索引擎，构建独立倒排索引，支持中文全文检索 |
| **缺乏原生 AI 能力** | 深度集成人工智能功能 | 在 Rust 后端统一调用 MiniMax API，所有 AI 功能内置，无需额外插件 |
| **PDF 元数据识别准确率低** | 大模型增强元数据提取 | 先提取 PDF 文本内容，再通过精心设计的提示词让 MiniMax 输出结构化元数据 |
| **官方云同步容量受限** | 提供自定义云同步选项 | 基于阿里云服务器搭建同步服务，支持数据库和附件的增量同步 |
| **插件系统复杂且不稳定** | 核心功能内置化 | 将常用功能直接写入应用核心，不依赖第三方插件系统 |

## 5. 技术栈与架构
### 5.1 技术栈选型
| 系统层级 | 技术选型 | 主要用途 |
|---------|---------|---------|
| **桌面应用外壳** | Tauri | 提供跨平台桌面应用能力，低内存高性能，使用系统原生 WebView |
| **前端框架** | React + TypeScript | 构建现代化、可维护的用户界面 |
| **UI 组件库** | Ant Design 或 Mantine | 提供丰富的现成组件，加速开发进度 |
| **前端状态管理** | Zustand 或 Jotai | 轻量级状态管理解决方案，避免过度复杂 |
| **PDF 渲染引擎** | pdf.js | 在 WebView 环境中可靠渲染 PDF 文档 |
| **PDF 标注处理** | Canvas API + pdf-lib（可选） | 实现标注绘制和导出功能 |
| **后端语言** | Rust | 处理性能敏感任务，保证内存安全和执行效率 |
| **数据存储** | SQLite（Zotero 原生） + Tantivy（搜索索引） | 主数据使用 Zotero 数据库，搜索功能使用独立的 Tantivy 索引 |
| **全文搜索** | Tantivy | Rust 原生高性能搜索引擎，支持中文分词 |
| **AI 服务集成** | MiniMax API | 提供摘要、翻译、问答、元数据提取等 AI 能力 |
| **前后端通信** | Tauri Command 系统 | 安全高效的前后端进程间通信机制 |
| **云同步服务** | 阿里云服务器 | 搭建自定义同步服务，存储数据库备份和附件文件 |

### 5.2 系统架构图
```mermaid
graph TD
    A[用户界面 - React] <-->|Tauri Command| B[Rust 后端核心]
    B --> C[SQLite 数据库 (Zotero 原生)]
    B --> D[Tantivy 全文搜索索引]
    B --> E[本地文件系统 (PDF 附件)]
    B --> F[MiniMax AI API]
    B --> G[阿里云同步服务]
    C --> H[Zotero Word 插件]
    E --> I[pdf.js PDF 预览组件]
```

## 6. 关键功能开发目标
### 6.1 Zotero 数据库访问层
- 实现跨平台的 Zotero 数据库路径自动检测功能
- 开发安全的数据库只读访问接口，避免破坏原始数据
- 实现文献基本信息（标题、作者、年份、摘要）的查询功能
- 实现文献分类（文件夹、标签）信息的读取和展示功能
- 设计安全的写操作机制，仅允许写入自定义字段或通过 Zotero API 进行修改

### 6.2 全文搜索系统
- 在 Rust 后端集成 Tantivy 搜索引擎
- 设计合理的索引结构，包含标题、作者、摘要、PDF 全文等字段
- 实现中文分词和搜索优化
- 开发增量索引更新机制，监听 Zotero 数据库变化并自动更新索引
- 实现毫秒级搜索响应和搜索结果高亮显示

### 6.3 PDF 预览与标注系统
- 集成 pdf.js 实现稳定的 PDF 文档预览功能
- 在 PDF 渲染层之上实现 Canvas 标注叠加层
- 开发多种标注工具：文本高亮、矩形、椭圆、箭头、自由绘制、文本注释
- 实现标注数据的序列化和持久化存储
- 支持标注的编辑、删除和导出功能

### 6.4 AI 功能集成
- 在 Rust 后端实现 MiniMax API 的统一调用接口
- 开发 AI 文献摘要生成功能，支持自定义摘要长度
- 实现文献标题和摘要的翻译功能
- 开发基于文献内容的自动标签生成功能
- 实现针对单篇文献的智能问答功能
- 开发 AI 元数据补全功能，提高 PDF 元数据识别准确率
- 添加 API 用量控制和错误处理机制

### 6.5 阿里云同步服务
- 设计基于阿里云服务器的同步协议
- 实现数据库文件的增量备份和恢复功能
- 开发 PDF 附件文件的同步功能
- 实现同步冲突检测和解决机制
- 提供同步状态显示和手动同步控制功能

### 6.6 用户界面与交互
- 设计现代化、简洁高效的主界面布局
- 实现文献列表视图和详细信息视图
- 开发左侧导航栏，支持文件夹和标签浏览
- 实现暗色/亮色主题切换
- 优化响应式布局，适配不同屏幕尺寸
- 添加必要的交互动画和加载状态提示

## 8. 注意事项与约束
### 8.1 数据安全
- 初期严格采用只读方式访问 Zotero 数据库，避免数据损坏风险
- 所有写操作必须经过严格测试，优先写入自定义字段
- 定期备份 Zotero 数据库和应用数据
- 云同步服务必须实现数据加密传输和存储

### 8.2 性能要求
- 应用启动时间控制在 2 秒以内
- 文献列表滚动流畅，无明显卡顿
- 全文搜索响应时间小于 100 毫秒
- 内存占用始终控制在 30MB 以内
- PDF 文档加载和渲染流畅

### 8.3 兼容性要求
- 保持与 Zotero 数据库结构的完全兼容
- 确保 Zotero Word 插件能够正常工作
- 支持 Windows、macOS 和 Linux 三大平台
- 兼容主流的 PDF 文档格式

### 8.4 第三方服务使用
- MiniMax API 调用必须添加合理的错误处理和重试机制
- 为 AI 功能添加用量控制，避免超出 API 限额
- 阿里云服务器资源使用需进行合理规划和监控
- 所有第三方服务依赖必须可配置，方便用户替换
